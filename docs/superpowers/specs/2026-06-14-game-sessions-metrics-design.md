# Game Sessions Metrics (CreatedAt and Length) Design Specification

This specification documents the changes made to the `GameSession` model and logic within the `game-service` to support multiple session documents per user, tracking both play frequency and duration (length) using an inactivity timeout.

---

## 1. Database Schema Changes

We will modify `packages/game-service/src/models/GameSession.js` to support multiple sessions and track session duration and state.

### Mongoose Schema Fields
We will add `status` and `length` to the `gameSessionSchema` definition:
*   `status`: `String`, enum `["active", "completed"]`, default `"active"`, indexed.
*   `length`: `Number`, default `0` (represents duration of active play in seconds).
*   `createdAt` and `updatedAt` are already auto-generated by the schema's `{ timestamps: true }` option.

### Indexes
We will replace the existing unique constraint `{ sessionKey: 1, gameId: 1 }` with a **partial unique index**. This guarantees that a child can only have at most one active session per game at any given time, while allowing any number of completed sessions.
*   **Remove**: `gameSessionSchema.index({ sessionKey: 1, gameId: 1 }, { unique: true });`
*   **Add**: `gameSessionSchema.index({ sessionKey: 1, gameId: 1 }, { unique: true, partialFilterExpression: { status: "active" } });`

## 2. Service Logic changes (`packages/game-service/src/services/gameService.js`)

We will introduce session lifecycle management to handle inactivity transitions.

### Inactivity Timeout Configuration
*   Define a constant `SESSION_TIMEOUT_MS = 15 * 60 * 1000` (15 minutes).

### Helper Function: `getActiveSession(gameId)`
This function will locate the current active session or handle its expiration:
1. Search for the session: `GameSession.findOne({ sessionKey: DEFAULT_SESSION_KEY, gameId, status: "active" })`.
2. If found, check if it has expired: `Date.now() - session.updatedAt > SESSION_TIMEOUT_MS`.
3. If it has expired:
    *   Set `session.status = "completed"`.
    *   Calculate the final length: `session.length = (session.updatedAt - session.createdAt) / 1000`.
    *   Set `session.activeQuestionId = null`.
    *   Save the completed session.
    *   Create a new active session document for the user.
4. If it has not expired, return it.
5. If no active session exists, create and return a new active session document.

### API Endpoint Adjustments

#### `GET /api/games/:gameId/session` (`fetchNextQuestion`)
*   Instead of doing a simple upsert via `findOneAndUpdate`, we will resolve the active session using the `getActiveSession` helper.
*   The active session's `activeQuestionId` is updated to the newly selected question ID.

#### `POST /api/games/:gameId/answer` (`submitAnswer`)
*   Retrieve the active session.
*   If the session expired (i.e. `Date.now() - session.updatedAt > SESSION_TIMEOUT_MS`), complete the session, set the length, save it, and return a `409 Conflict` (or similar) error indicating that the session has expired.
*   If valid:
    *   Push the answered question to `answeredQuestions`.
    *   Increment the score.
    *   Set `activeQuestionId` to `null`.
    *   Recalculate `length` as `(Date.now() - session.createdAt) / 1000`.
    *   Save the session document.
